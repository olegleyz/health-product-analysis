## Milestone 1: Foundation + Data Pipes

**Goal:** A working Telegram bot that can converse about health AND has real device data flowing in. Validates: LLM pipeline, tone, data sync from Garmin/Oura/Renpho/Strava.

**Why data is in M1:** The concierge's awareness of real data (sleep, training, weight) is core to the product, not an enhancement. A concierge that asks "did you work out?" when Garmin already has the data feels broken.

**Deliverables:**

1. **Project setup**
   - Python project (poetry/uv for deps)
   - SQLite database with schema
   - Config file (API keys, user IDs, schedules)

2. **Telegram bot (reactive)**
   - Receives messages via long polling
   - Sends responses via Telegram API
   - Uses python-telegram-bot library

3. **Concierge Brain (reactive path)**
   - Prompt builder with persona definition
   - Claude API integration (anthropic Python SDK)
   - Basic safety filter
   - Lightweight data extractor

4. **Data sync scripts**
   - Garmin Connect sync (`garminconnect` lib): activities, sleep, stress, Body Battery, steps
   - Oura Ring sync (REST API): sleep, readiness, HRV, activity
   - Renpho sync (Apple Health export or reverse-engineered API): weight, body composition
   - Strava sync (`stravalib`): activities with detailed metrics
   - All write to normalized `device_data` table in SQLite
   - Cron job: runs every 4 hours

5. **Onboarding conversation**
   - Goals, routine, preferences, tone calibration
   - Stores profile in SQLite

**Technical tasks:**
- [ ] Initialize Python project with dependencies
- [ ] Create SQLite schema (users, messages, device_data, engagement_state)
- [ ] Implement Telegram bot with long polling
- [ ] Design persona prompt (system prompt, tone examples, anti-examples)
- [ ] Implement prompt builder (context from profile + recent messages + device data)
- [ ] Implement Claude API client
- [ ] Implement basic safety filter (keyword rules)
- [ ] **Spike: Garmin Connect data extraction** — test `garminconnect` lib, verify what data is available, handle auth
- [ ] **Spike: Oura API data extraction** — get personal access token, test endpoints
- [ ] **Spike: Renpho data extraction** — test reverse-engineered API or Apple Health export path
- [ ] **Spike: Strava API data extraction** — OAuth2 flow, test activity pull
- [ ] Implement data sync scripts for each source
- [ ] Set up cron jobs for data sync
- [ ] Implement onboarding flow
- [ ] Test: send a message, get a response that references today's Garmin/Oura data
- [ ] Deploy: run on local Mac or VPS with systemd/launchd

**Exit criteria:**
- Bot responds coherently to health messages
- Device data from all 4 sources is syncing automatically
- Concierge can reference real data: "I see you ran 5K this morning on your Garmin"
- Onboarding stores profile correctly
- Runs unattended for 48 hours without crashing

## Milestone 2: Proactive Concierge

**Goal:** The concierge reaches out first. Morning check-ins that reference last night's sleep data. Post-workout follow-ups. Bedtime nudges. This is where it becomes the product.

**Deliverables:**

1. **Scheduled check-ins (cron-based)**
   - Morning check-in: references last night's sleep (Oura), today's readiness, any planned activities
   - Evening check-in: reviews the day — training done, meals, sleep intention
   - Pre-generated by Concierge Brain using current context + device data

2. **Contextual nudges**
   - Post-activity: "Nice 10K! Make sure you eat well and hydrate"
   - Bedtime: based on stated goal + recent sleep trends from Oura
   - Drift: no workout in X days, sleep getting worse, weight trending up

3. **Frequency governor**
   - Max 4 outbound messages/day
   - Backoff when messages go unanswered
   - Quiet mode after extended silence
   - No messages before 7 AM or after 11 PM

4. **Engagement state tracking**
   - Active / Quiet / Paused states
   - Transitions based on user responsiveness

**Technical tasks:**
- [ ] Implement morning check-in script (cron, 8:00 AM)
- [ ] Implement evening check-in script (cron, 9:00 PM)
- [ ] Extend prompt builder: include device data summary in proactive context
- [ ] Implement frequency governor (SQLite-backed counters)
- [ ] Implement engagement state machine
- [ ] Implement drift detection (simple rules on device_data: workout gap, sleep trend, weight trend)
- [ ] Implement nudge triggers: post-activity (detect new activity in device_data), bedtime, drift
- [ ] Test: simulate a full day — morning check-in with sleep data → user replies → evening check-in
- [ ] Test: silence handling — user doesn't reply for 2 days → quiet mode
- [ ] Run for 1 week with both users

**Exit criteria:**
- Morning message arrives with accurate sleep/readiness data
- Post-workout messages reference the actual workout from Garmin/Strava
- Frequency caps work — never more than 4 messages/day
- Both users say: "This feels like someone checking in on me, not spam"
- Runs for 7 days unattended

## Milestone 3: Intelligence + Nutrition

**Goal:** The concierge gets smarter. Daily summaries, long-term pattern awareness, and a meal memory that enables nutrition recommendations.

**Deliverables:**

1. **Daily summaries**
   - End-of-day LLM job: summarizes conversation + device data into structured summary
   - Stored for long-term context (prompt builder uses last 7–14 days of summaries)

2. **Enhanced personalization**
   - References past weeks: "Your sleep has been improving this week"
   - Connects patterns: "You tend to sleep worse after late dinners"
   - Adapts check-in content based on what matters most right now

3. **Meal memory & recommendations**
   - Extracts meals from conversation → stores in `meals` table
   - Builds user's meal repertoire over time
   - Recommends meals based on context:
     - Training day: "How about that chicken rice bowl you had last Tuesday?"
     - Rest day: "Good day for something lighter — you mentioned liking that big salad"
     - Drift: "You haven't cooked in a while. What about that salmon recipe?"
   - Suggests gradual adjustments to existing meals

4. **Conversation compression**
   - Messages older than 7 days compressed into summaries
   - Keeps context window manageable

**Technical tasks:**
- [ ] Implement daily summary generation (cron, end of day)
- [ ] Design summary prompt (conversation + device data → structured output)
- [ ] Extend prompt builder to include 14-day summary history
- [ ] Implement meal extraction from conversations
- [ ] Create meals table and meal memory logic
- [ ] Design nutrition recommendation prompts (uses meal table + context)
- [ ] Implement conversation compression job
- [ ] Test: verify concierge references past week accurately
- [ ] Test: verify meal recommendations draw from actual user meals
- [ ] Run for 2 weeks with both users

**Exit criteria:**
- Daily summaries are accurate
- Concierge naturally references patterns from past weeks
- Meal recommendations reference actual meals the user has eaten
- Users feel the concierge "knows them" after 2 weeks
- Context window stays within Claude's limits

## Milestone 4: Polish & Iterate

**Goal:** Tune everything based on real usage. Fix what's annoying, improve what works, add quality-of-life features.

**Deliverables (driven by real feedback):**

Likely areas:
- Tone adjustments (too formal? too casual? too frequent?)
- Timing adjustments (check-in times, nudge triggers)
- Better use of device data (which metrics matter most?)
- Smarter drift detection (reduce false positives)
- Weekly/monthly health reflections
- Lab result interpretation (upload PDF, get context-aware analysis)
- Better meal recommendations as repertoire grows

**Technical tasks:**
- [ ] Collect 2 weeks of feedback from both users
- [ ] Adjust persona prompt based on feedback
- [ ] Tune check-in timing and frequency
- [ ] Add weekly reflection (Sunday evening summary of the week)
- [ ] Improve drift detection thresholds
- [ ] Add any missing device data fields that turned out to matter
- [ ] Consider WhatsApp migration if Telegram is inconvenient

**Exit criteria:**
- Both users are actively using the concierge daily for 30+ days
- No "annoying" messages reported in the last 2 weeks
- Concierge feels personal and context-aware

---

## Testing Strategy (Keep It Simple)

For 2 users, heavy test infrastructure is waste. Instead:

- **Manual testing:** Try every feature yourself before deploying
- **Conversation review:** Read through generated messages weekly. Are they good?
- **Device data spot-check:** Verify synced data matches what the watch/ring/scale shows
- **Safety spot-check:** Occasionally try to get the concierge to give medical advice. It should refuse.
- **Logs:** Keep a log of every outbound message. Review when something feels off.
